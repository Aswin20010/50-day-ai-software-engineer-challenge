# Day 32 — UI Refactor Plan
08
## Purpose

The purpose of Day 32 is to refactor the existing API Factory UI into smaller, maintainable components without changing the current behavior.

The current `src/app/page.tsx` file has grown large because it contains multiple responsibilities in one file:

* Requirement file upload
* Requirement extraction API call
* Missing fields display
* Normalized extraction review
* Open questions and gaps display
* Manual Node-RED rule builder
* Flow generation API call
* Generation metadata display
* Adapter warning display
* Generated flow JSON display
* Download JSON logic

Day 32 will split this page into focused components so the UI becomes easier to maintain and extend.

---

## Day 32 Goal

Refactor the large page into smaller files while preserving all existing functionality from Day 27 through Day 31.

This is a structural refactor only.

No business behavior should change.

---

## Important Refactor Rule

The main rule for Day 32 is:

```text
Do not change behavior.
```

This means:

* Do not change API endpoints.
* Do not change request payloads.
* Do not change response parsing.
* Do not change field names.
* Do not change generated flow structure.
* Do not change normalized extraction handling.
* Do not change adapter metadata handling.
* Do not change download behavior.
* Do not replace the existing page with a different workflow.

The UI should work the same after the refactor as it worked before the refactor.

---

## Current Page Responsibilities

The current `src/app/page.tsx` handles the full API Factory workflow:

```text
Upload Confluence file
→ Extract requirements
→ Review normalized extraction
→ Review gaps and open questions
→ Edit generated business rules
→ Generate Node-RED flow
→ Review generation metadata
→ Inspect generated rule coverage
→ Review adapter warnings
→ Download generated-flow.json
```

This workflow should remain unchanged.

---

The Day 32 refactor will separate the current page into four layers:

## What Should Not Move Yet

For Day 32, keep the main API call functions inside `page.tsx` initially:

```text
extractRequirementsFromFile
generateFlow
downloadFlow
```

Reason:

Moving API logic and UI rendering at the same time increases risk.

Day 32 should focus on safely extracting UI components first.

A later day can move API actions into service files.

## Risk Management

Refactoring can easily break a working UI.

To reduce risk:

1. Move types first.
2. Move helpers second.
3. Move read-only display components next.
4. Move interactive rule builder components after that.
5. Update `page.tsx` last.
6. Run `npm run build` after each major phase.
7. Avoid changing API payloads.
8. Avoid changing backend endpoints.
