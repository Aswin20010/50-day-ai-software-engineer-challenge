# 27. Portfolio Case Study – Business Problem

```text
Enterprise API work often begins with incomplete requirement documents and
ends with manual reviews of legacy codebases.

For new APIs, engineers must manually identify endpoints, request fields,
responses, validations, business rules, and missing information.

For existing APIs, architects must inspect source files, identify implemented
capabilities, compare the system with a target architecture, document risks,
and translate findings into a migration backlog.

These activities are slow, difficult to standardize, and hard to audit when
findings are not linked to evidence.
```

---

# 28. Portfolio Case Study – Solution

```text
The platform separates the modernization lifecycle into two complementary
workflows.

The API Factory creates structured specifications from unstructured
requirements.

The Migration Analyzer creates structured modernization plans from existing
API artifacts.

Both workflows use typed domain models, deterministic processing, explicit
validation, stable identifiers, and browser-based review.
```

# 29. Key Engineering Decisions

## 29.1 AI Output Is Not Final Output

```text
AI-assisted extraction produces candidate data. Normalization, validation,
missing-information detection, and human review occur before the result is
treated as usable.
```

## 29.2 Evidence Before Findings

```text
The Migration Analyzer extracts endpoint and capability evidence before
generating gaps or risks. Findings retain links to their source artifacts.
```

## 29.3 Explicit Target Architecture

```text
The analyzer does not evaluate a project against an undefined idea of
“modern.” It uses configurable target architecture rules with explicit
categories, severities, weights, and evidence expectations.
```

## 29.4 Deterministic Downstream Pipeline

```text
Once structured evidence is available, rule evaluation, deduplication,
finding generation, recommendation generation, task generation, and scoring
are designed to remain deterministic.
```

## 29.5 Human Approval

```text
The platform is designed as engineering decision support. It does not
independently approve API requirements or migration plans.
```

# 31. Challenge–Action–Result Interview Format

## Example 1 – State Refactor

```text
Challenge:
The extraction page held too much workflow state and reset logic, making the
component difficult to maintain and test.

Action:
I moved extraction behavior into a reusable custom hook and extracted reset
state construction into typed helper functions.

Result:
The page became more focused on presentation, workflow behavior became
reusable, and state transitions became easier to test.
```

## Example 2 – Analyzer Traceability

```text
Challenge:
Architecture findings could become generic or difficult to verify when they
were generated without a stable source model.

Action:
I introduced source artifact and evidence models, then required rule results,
gaps, risks, recommendations, and migration tasks to preserve linked
identifiers.

Result:
The UI could display findings together with the evidence used to support them.
```

## Example 3 – Validation Script Failure

```text
Challenge:
The final validation script failed because lint warnings were configured to
block completion.

Action:
I reviewed script files, removed unused imports, separated required and
optional validation stages, and preserved nonzero exit codes.

Result:
The validation process became stricter and its final status became easier to
interpret.
```

# 37. System Design Tradeoffs

## Tradeoff 1 – Structured Artifacts Instead of Raw Repository Parsing

```text
The prototype uses supplied source artifacts, which makes evaluation and
testing deterministic but limits automatic analysis of full repositories.
```

## Tradeoff 2 – Rule-Based Scoring

```text
Explicit rules improve explainability and reproducibility, but the scoring
model depends on the selected target architecture.
```

## Tradeoff 3 – Human Review

```text
Human review prevents unsupported decisions, but it means the system is a
decision-support tool rather than a fully autonomous modernization platform.
```

## Tradeoff 4 – Browser Prototype

```text
Next.js enabled a fast end-to-end review workflow, but production concerns such
as authentication, persistence, multi-user collaboration, and deployment were
outside the prototype scope.
```

# 38. Questions a Recruiter May Ask

## What was the most difficult part?

Recommended answer:

```text
The most difficult part was preserving explainability across the complete
analysis chain. It was not enough to generate a gap or recommendation. I
needed stable relationships from source artifacts to evidence, rule outcomes,
gaps, risks, recommendations, and tasks so that users could verify why each
result existed.
```

## Why not use only an LLM?

```text
LLMs are useful for interpreting unstructured text, but they may return
inconsistent or unsupported output. I used AI as an assistive stage and kept
normalization, schema validation, rule evaluation, scoring, and traceability
deterministic.
```

## Is this production ready?

```text
It is a working prototype with modular architecture, tests, validation, sample
data, and browser workflows. Production readiness would require authentication,
persistent storage, security review, deployment automation, runtime
observability, expanded evaluation, and stabilized end-to-end testing.
```

## How did you test it?

```text
I created unit tests for core extraction and analysis modules, form and request
validation, evidence helpers, deduplication, orchestration, and browser-domain
logic. I also used strict linting, TypeScript validation, production builds,
and manual browser workflow validation. Playwright remains a documented area
for stabilization where applicable.
```

# 39. Questions a Hiring Manager May Ask

## How is the analyzer extensible?

```text
Target architectures, architecture rules, evidence categories, severity
values, and task-generation behavior are represented through typed data and
separate modules. New architectures or rules can be added without rewriting
the browser workflow.
```

## How do you prevent duplicate findings?

```text
I separate extraction and generation stages and apply normalized identifiers
and deduplication helpers to evidence, gaps, and risks before generating
downstream recommendations and tasks.
```

## How do you handle incomplete information?

```text
The API Factory preserves missing and ambiguous information as structured
output rather than silently inventing values. The user can review and resolve
those items before using the specification.
```

## How do you calculate readiness?

```text
Each architecture rule has an explicit category, outcome, and weight. Category
scores are calculated from applicable rule outcomes, and the overall score is
derived from weighted category scores. Findings remain linked to the rules and
evidence behind the score.
```
