# Day 11 — API Contract Mapping Rules

## Purpose

This document defines how API-level requirement details should be mapped into Workflow Schema v2.

The API contract is the public interface of an API. It describes how clients call the API, what input the API expects, and what response the API returns.

This mapping file focuses on converting API contract information from raw requirements into Schema v2 fields.

No coding is performed in this phase.

---

## Background

On Day 10, Workflow Schema v2 introduced a formal `api_contract` block and supporting blocks such as `field_validations`, `response_mappings`, `error_mappings`, and `source_traceability`.

Day 11 defines how raw requirements should populate those blocks.

API contract information may come from:

* Confluence requirement documents
* API design documents
* OpenAPI-style contracts
* Request and response examples
* Node-RED HTTP In nodes
* Existing controller or route notes
* Excel intake expectations
* User-confirmed clarifications

---

## API Contract Mapping Objective

The objective of API contract mapping is to identify the public API details and place them into the correct Schema v2 sections.

API contract details should answer:

```text
What endpoint is exposed?
Which HTTP method is used?
What version does the API belong to?
What headers are expected?
What request body does the API accept?
What response does the API return?
What status codes can be returned?
What operation name should represent this endpoint?
```

---

# Schema v2 Blocks Used

API contract mapping primarily uses these Schema v2 blocks:

```text
api_contract
field_validations
response_mappings
error_mappings
source_traceability
generator_readiness
```

Some API contract details may also influence:

```text
workflow_steps
test_scenarios
```

For example, required headers create validation rules and negative test scenarios.

# 1. Endpoint Path Mapping

## Requirement Signal

Look for phrases such as:

```text
Endpoint:
API URL:
Path:
Route:
Resource:
POST /api/v1/fabrics
GET /api/v1/accounts/{accountId}
The service exposes /api/v1/discount/evaluateTransactionDiscount
```

## Mapping Rule

Map the endpoint path to:

```yaml
api_contract.path
```

## Example Requirement

```text
The API shall expose POST /api/v1/fabrics.
```

## Schema v2 Mapping

```yaml
api_contract:
  path: /api/v1/fabrics
```

## Notes

The path should include the full route from the public API root.

If the requirement mentions only a partial route, map the known value and mark confidence as medium or low in `source_traceability`.

# 2. HTTP Method Mapping

## Requirement Signal

Look for HTTP method words:

```text
GET
POST
PUT
PATCH
DELETE
```

Also look for route examples such as:

```text
POST /api/v1/fabrics
```

## Mapping Rule

Map the method to:

```yaml
api_contract.method
```

## Example Requirement

```text
POST /api/v1/discount/evaluateTransactionDiscount
```

## Schema v2 Mapping

```yaml
api_contract:
  method: POST
```

## Notes

If no HTTP method is explicitly stated, infer only if the request example strongly proves it.

Otherwise, mark the method as unknown and add a low-confidence traceability note.

---

# 3. Endpoint ID Mapping

## Requirement Signal

Endpoint IDs may not exist in raw requirements.

They are usually generated by the mapping process.

## Mapping Rule

Create a stable endpoint ID using the pattern:

```text
EP-<DOMAIN>-<NUMBER>
```

Examples:

```text
EP-DISCOUNT-001
EP-FABRIC-001
EP-ACCOUNT-001
```

Map it to:

```yaml
api_contract.endpoint_id
```

## Schema v2 Mapping

```yaml
api_contract:
  endpoint_id: EP-FABRIC-001
```

## Notes

Endpoint IDs should be unique within the project.

If the workflow has only one endpoint, use `001`.

---

# 4. API Version Mapping

## Requirement Signal

Look for version in:

```text
/api/v1/
version v1
API version: 1
requestorinfo-version
```

## Mapping Rule

Map public API version to:

```yaml
api_contract.api_version
```

## Example Requirement

```text
POST /api/v1/fabrics
```

## Schema v2 Mapping

```yaml
api_contract:
  api_version: v1
```

## Notes

Do not confuse public API version with header version.

For example:

```text
/api/v1/fabrics
```

maps to `api_contract.api_version`.

A header like:

```text
requestorinfo-version
```

maps to `field_validations`, not `api_contract.api_version`.

# 5. Operation Name Mapping

## Requirement Signal

Look for phrases such as:

```text
Operation:
Operation name:
Service operation:
Use case:
Function:
API name:
```

If the requirement does not provide a name, derive one from the route.

## Mapping Rule

Map operation name to:

```yaml
api_contract.operation_name
```

## Naming Convention

Use lower camel case.

Examples:

```text
evaluateTransactionDiscount
getFabrics
createApplicationMessage
interrogateAccount
```

## Example Requirement

```text
The API evaluates transaction discount eligibility.
```

## Schema v2 Mapping

```yaml
api_contract:
  operation_name: evaluateTransactionDiscount
```

## Notes

If operation name is inferred from the path, mark source confidence as medium.

# 6. Summary Mapping

## Requirement Signal

Look for purpose or description statements such as:

```text
This API is used to...
The purpose of this service is...
This endpoint returns...
The API validates and processes...
```

## Mapping Rule

Map a concise summary to:

```yaml
api_contract.summary
```

## Example Requirement

```text
This API validates SKU numbers and fabric type, calls the fabric customization service, and returns fabric configuration.
```

## Schema v2 Mapping

```yaml
api_contract:
  summary: Validate SKU numbers and fabric type, call fabric customization service, and return fabric configuration.
```

## Notes

The summary should be one sentence.

Detailed workflow behavior belongs in `workflow_steps`, not the API summary.

# 7. Content Type Mapping

## Requirement Signal

Look for:

```text
Content-Type: application/json
Consumes: application/json
Request body: JSON
```

## Mapping Rule

Map request content type to:

```yaml
api_contract.content_type
```

## Default Rule

If the API has a JSON request body and no other content type is stated, use:

```text
application/json
```

but mark confidence as medium if not explicitly stated.

## Schema v2 Mapping

```yaml
api_contract:
  content_type: application/json
```

# 12. Header Mapping

## Requirement Signal

Look for header tables or phrases such as:

```text
Required headers:
Headers:
Client-id is required
Division-Id must be provided
Trace-Id is optional
```

## Mapping Rule

Headers should map to `field_validations`, not only `api_contract`.

Each header should become one field validation entry.

## Required Header Mapping

```yaml
field_validations:
  - validation_id: VAL-FABRIC-HDR-001
    field_path: request.headers.Division-Id
    location: header
    type: integer
    required: true
    failure_behavior:
      http_status: 400
      error_code: BAD_REQUEST
      message: Division-Id header is required.
```

## Optional Header Mapping

```yaml
field_validations:
  - validation_id: VAL-FABRIC-HDR-002
    field_path: request.headers.Trace-Id
    location: header
    type: string
    required: false
```

## Notes

Headers are part of the API contract, but Schema v2 stores validation details in `field_validations`.

---

# 13. Body Field Mapping

## Requirement Signal

Look for request body tables, JSON examples, or phrases such as:

```text
Request body:
Payload:
Body fields:
skuNumbers is required
fabricTypeId is required
account.encryptInfo.encryptedData is required
```

## Mapping Rule

Each body field should map to `field_validations`.

Nested body fields should use dot notation.

Array item validation should use `[]`.

## Example Requirement

```json
{
  "skuNumbers": ["SKU123", "SKU456"],
  "fabricTypeId": 3
}
```

## Schema v2 Mapping

```yaml
field_validations:
  - validation_id: VAL-FABRIC-BODY-001
    field_path: request.body.skuNumbers
    location: body
    type: array
    required: true
    min_items: 1

  - validation_id: VAL-FABRIC-BODY-002
    field_path: request.body.skuNumbers[]
    location: body
    type: string
    required: true
    allow_blank: false

  - validation_id: VAL-FABRIC-BODY-003
    field_path: request.body.fabricTypeId
    location: body
    type: integer
    required: true
```

# 14. Path Parameter Mapping

## Requirement Signal

Look for route variables such as:

```text
/api/v1/accounts/{accountId}
/api/v1/stores/{storeNbr}/items/{itemId}
```

## Mapping Rule

Path parameters should map to `field_validations`.

Use location:

```text
path
```

## Example

```yaml
field_validations:
  - validation_id: VAL-ACCOUNT-PATH-001
    field_path: request.path.accountId
    location: path
    type: string
    required: true
```

## Notes

The path itself still maps to `api_contract.path`.

The variable inside the path maps to `field_validations`.

# 16. Success Status Mapping

## Requirement Signal

Look for:

```text
Returns 200 OK
Success response: 201 Created
HTTP 204 on success
```

## Mapping Rule

Success status should map to response mappings.

Use:

```yaml
response_mappings.http_status
```

## Example

```yaml
response_mappings:
  - mapping_id: RESP-FABRIC-001
    response_type: success
    http_status: 200
```

## Notes

If no success status is provided, use common REST behavior cautiously:

* `200` for successful read/process response
* `201` for creation
* `204` for success with no body

Mark confidence as medium or low if inferred.

---

# 17. Error Status Mapping

## Requirement Signal

Look for:

```text
400 Bad Request
401 Unauthorized
404 Not Found
422 Business rule failure
500 Internal Server Error
502 Downstream service error
```

## Mapping Rule

Error statuses should map to:

```yaml
error_mappings
```

## Example

```yaml
error_mappings:
  - error_mapping_id: ERR-VALIDATION-001
    error_source: validation
    condition: required field missing
    target_error_code: BAD_REQUEST
    target_http_status: 400
    target_message: Required field is missing.
    retryable: false
    fail_fast: true
```

---

# 18. Success Response Body Mapping

## Requirement Signal

Look for response examples or field mapping tables.

Examples:

```json
{
  "discountType": "Employee",
  "discountPercentApplied": 20
}
```

or:

```text
Map calculatedDiscount.discountType to response.discountType.
```

## Mapping Rule

Success response fields map to:

```yaml
response_mappings
```

## Example

```yaml
response_mappings:
  - mapping_id: RESP-DISCOUNT-001
    mapping_name: Map Discount Type
    applies_to_step: STEP-012
    response_type: success
    http_status: 200
    source: calculatedDiscount.discountType
    target: response.discountType
    transformation: direct_copy
    required: true
```

## Notes

If the response example does not specify source fields, map the target response field and mark source as `unknown` or inferred.

```yaml
error_mappings
```

# 20. Request Example Mapping

## Requirement Signal

Look for sample curl commands, JSON examples, or sample payloads.

## Mapping Rule

Request examples are not the main source of schema truth, but they can support:

```text
field_validations
test_scenarios
source_traceability
```

## Example

A sample request can become a positive test scenario:

```yaml
test_scenarios:
  - scenario_id: TC-FABRIC-001
    scenario_name: Valid Fabric Request
    scenario_type: positive
    input:
      headers:
        Division-Id: 12
      body:
        skuNumbers:
          - SKU123
        fabricTypeId: 3
```

---

# 21. Response Example Mapping

## Requirement Signal

Look for success or error response examples.

## Mapping Rule

Response examples support:

```text
response_mappings
error_mappings
test_scenarios
```

## Notes

A response example should not be blindly treated as the full response contract unless the requirement says it is complete.

If uncertain, mark confidence as medium.

# 22. API Contract Source Traceability

Every mapped API contract item should include traceability when possible.

## Mapping Rule

Source references should map to:

```yaml
source_traceability
```

## Example

```yaml
source_traceability:
  workflow_source:
    requirement_doc: Fabric API Requirement
    requirement_section: API Contract
    source_text_reference: Section 2.1
    confidence: high

  field_sources:
    - field_path: request.headers.Division-Id
      requirement_section: Header Requirements
      source_text_reference: Section 2.2
      confidence: high
```

# Example 1 — Fabric API Contract Mapping

## Raw Requirement

```text
POST /api/v1/fabrics

Headers:
- Division-Id required, allowed values 12 and 13
- Trace-Id optional

Request body:
- skuNumbers required, non-empty array of strings
- fabricTypeId required integer

Success response: 200
```

## Schema v2 Mapping

```yaml
api_contract:
  endpoint_id: EP-FABRIC-001
  path: /api/v1/fabrics
  method: POST
  api_version: v1
  content_type: application/json
  response_type: application/json
  operation_name: getFabrics
  summary: Validate fabric request, call fabric customization service, and return fabric data.
  tags:
    - fabric
    - customization

field_validations:
  - validation_id: VAL-FABRIC-HDR-001
    field_path: request.headers.Division-Id
    location: header
    type: integer
    required: true
    allowed_values:
      - 12
      - 13
    failure_behavior:
      http_status: 400
      error_code: INVALID_DIVISION
      message: Division-Id must be 12 or 13.

  - validation_id: VAL-FABRIC-HDR-002
    field_path: request.headers.Trace-Id
    location: header
    type: string
    required: false

  - validation_id: VAL-FABRIC-BODY-001
    field_path: request.body.skuNumbers
    location: body
    type: array
    required: true
    min_items: 1

  - validation_id: VAL-FABRIC-BODY-002
    field_path: request.body.skuNumbers[]
    location: body
    type: string
    required: true
    allow_blank: false

  - validation_id: VAL-FABRIC-BODY-003
    field_path: request.body.fabricTypeId
    location: body
    type: integer
    required: true

response_mappings:
  - mapping_id: RESP-FABRIC-001
    mapping_name: Map Fabric Success Response
    response_type: success
    http_status: 200
    source: downstreamFabricResponse.body
    target: response.body
    transformation: direct_copy
    required: true
```

# Mapping Review Checklist

Use this checklist after mapping API contract details.

```text
- Is endpoint path captured?
- Is HTTP method captured?
- Is API version captured?
- Is operation name defined?
- Is request content type captured?
- Is response content type captured?
- Are required headers mapped to field_validations?
- Are optional headers mapped to field_validations?
- Are body fields mapped to field_validations?
- Are path parameters mapped if present?
- Are query parameters mapped if present?
- Are success statuses mapped?
- Are error statuses mapped?
- Are response fields mapped?
- Are request and response examples converted into test scenario candidates?
- Is source traceability captured?
- Are inferred values marked with appropriate confidence?
```

# Common Mistakes to Avoid

## Mistake 1 — Putting All Fields Only in api_contract

Headers and body fields should not live only in `api_contract`.

They should be represented through `field_validations`.

## Mistake 2 — Ignoring Optional Fields

Optional fields must still be captured.

They may influence request mapping, downstream pass-through, response mapping, or documentation.

## Mistake 3 — Treating Examples as Complete Contracts

Request and response examples may be incomplete.

Use them as evidence, but mark confidence appropriately.

## Mistake 4 — Confusing API Version with Header Version

A path like `/api/v1/...` maps to `api_contract.api_version`.

A header like `requestorinfo-version` maps to `field_validations`.

## Mistake 5 — Dropping Error Statuses

Every explicit error status should map to `error_mappings`.

## Mistake 6 — Missing Source Traceability

Every major contract decision should trace back to a requirement source.

---

# Final Summary

API contract mapping defines how public API interface details move from raw requirements into Workflow Schema v2.

The main target block is `api_contract`, but API contract mapping also fills:

* `field_validations`
* `response_mappings`
* `error_mappings`
* `test_scenarios`
* `source_traceability`

This mapping step is important because it creates a clear public API contract before workflow logic, downstream services, database rules, and generation behavior are added.

This completes Phase 2 of Day 11.
