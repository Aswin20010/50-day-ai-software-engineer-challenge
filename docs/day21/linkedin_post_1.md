## Refined LinkedIn Post

For the past 20 days, I have been working on a focused challenge:

**Can AI help automate enterprise API development starting from requirement documents?**

In many enterprise teams, API development does not begin with code.

It begins with scattered information across:

- Confluence pages
- Business rules
- Request and response examples
- Downstream API details
- Database mappings
- Error handling expectations
- Discussions between product, business, and engineering teams

The hard part is not just writing the API.

The hard part is understanding the requirement clearly enough to build the right API.

That is the problem I am exploring through my **50-Day AI Software Engineer Challenge**.

The goal is to design a system that can take enterprise API requirements and convert them into structured engineering artifacts such as:

- Workflow representations
- Node-RED flows
- Excel intake specifications
- Gap analysis reports
- API generation inputs
- Eventually, Go service structures

The research direction behind this work is:

**Automating Enterprise API Development using AI-Driven Requirement and Workflow Synthesis.**

During the first 20 days, I focused on the foundation rather than jumping directly into code.

I worked on:

- Defining the research problem
- Designing the system architecture
- Studying existing approaches
- Creating dataset and evaluation strategies
- Designing requirement extraction specifications
- Planning workflow representation
- Defining schema and validation concepts
- Structuring how LLMs can transform messy requirements into usable engineering artifacts

One important lesson so far:

**AI code generation is only useful when the requirement understanding is strong.**

If the input requirement is unclear, incomplete, or inconsistent, the generated code will also be unreliable.

That is why this project focuses on the layers before code generation:

1. Requirement extraction
2. Workflow synthesis
3. Schema validation
4. Artifact generation
5. Evaluation and gap analysis

The bigger idea is that an AI Software Engineer should not only generate code.

It should also help understand requirements, identify missing information, model workflows, validate assumptions, and create reviewable artifacts for engineering teams.

This is still a work in progress, but the first 20 days have helped me see the problem more clearly.

Reliable AI-assisted software engineering starts before code.

It starts with better requirement understanding.
